# Skill to Web Prompt

将 Claude Code 技能转化为网页 AI 可用的提示词，并给出诚实的转化分析。

## 能力差异

Claude Code 有而网页 AI 没有的能力，以及对应的转换策略：

| 能力 | 转换策略 |
|------|---------|
| 执行脚本 (scripts/) | 简单逻辑用自然语言描述融入提示词；复杂逻辑可在提示词中附上脚本代码，标注"请手动运行" |
| 读写文件 (Write/Edit/Read) | 提示词中让 AI 输出内容，用户自行复制保存或粘贴提供 |
| 运行命令 (Bash) | 转为用户手动执行的步骤说明，在提示词中给出完整的命令文本 |
| 搜索代码 (Glob/Grep) | 无法替代，在分析中诚实告知，并在使用说明中让用户自行找到文件内容 |
| 多步骤编排 | 拆解为 AI 可逐步协助的交互步骤 |
| 子代理 (Agent) | 无法替代，如果是核心功能则如实标记为丢失 |

网页 AI 天然擅长的：文本理解和生成、代码编写和解释、内容改写/翻译/总结、分析和建议。

## 转换流程

### 1. 读取技能

技能路径：`.agents/skills/<name>/`（符号链接指向此路径）。读取以下文件：
- `SKILL.md`（必须）
- `references/` 和 `scripts/` 下的所有文件
- 留意 frontmatter 中的 `allowed-tools`

### 2. 理解技能

在开始转换之前，先彻底理解这个技能：

- **核心价值**：用户为什么需要这个技能？它解决了什么痛点？
- **核心机制**：它是怎么做到的？纯知识指导？还是依赖工具编排？
- **工具依赖**：核心功能对 Bash、文件操作、子代理等的依赖程度
- **用户视角**：用户在 Claude Code 中使用这个技能时，体验是什么样的？

理解透了再动手转换。不要看到一个技能就直接套模板。

### 3. 生成提示词

提示词是用户拿去用的核心产物。质量优先级高于分析报告。

**提示词必须做到：**
- **自包含**：复制即用，不引用"原技能"或"脚本文件"。所有规则、示例、参考知识都内嵌其中
- **有角色**：开头设定角色（"你是一位……"），让网页 AI 进入正确的行为模式
- **可执行**：给出明确的步骤，不是模糊的方向。用户复制粘贴后，网页 AI 应该清楚知道要做什么
- **不伪装能力**：不要在提示词中要求网页 AI "读取文件"或"执行命令"。如果某步骤需要用户手动操作，在提示词中明确说出来

**转换时注意：**
- 纯知识/规则/方法论 → 直接融入提示词，几乎无损
- 脚本逻辑 → 简单的用自然语言描述；复杂的附上完整代码，标注"如果需要运行此脚本，请将代码复制到本地执行"
- 文件操作 → 改为交互式："请将需要处理的文本粘贴在下方""请将上述结果复制保存"
- 工具编排 → 拆为清晰的步骤序列，每步标注"AI 做 X，你（用户）做 Y"
- Claude Code 专属编排逻辑（如自动 git 操作、自动部署等）→ 从提示词中删除，在分析中说明丢失

**提示词长度：** 根据技能复杂度灵活把握。文本处理类技能可以把规则写全（3000-8000 字都可以），工具编排类技能精简到核心方法论（1000-3000 字）。原则是：够用但不冗余。

### 4. 评估实现度

| 实现度 | 判定标准 |
|--------|---------|
| 90-100% | 纯文本/代码处理技能，核心零依赖工具 |
| 70-89% | 核心为文本/代码处理，少量文件操作可手动替代 |
| 50-69% | 核心功能部分依赖工具，但主要逻辑可迁移 |
| 20-49% | 大量依赖工具，仅框架和概念可迁移 |
| 0-19% | 几乎完全依赖 Claude Code 独有能力，基本无法迁移 |

评估时，考虑的是**核心功能**的可迁移程度，不是所有子功能的算数平均。如果一个技能有 20 个子功能但其中 2 个占了 80% 的价值，那两个的迁移情况就决定了整体实现度。

## 输出

你的输出应该是一份完整的文档，让用户能看懂、能用、能理解差异。以下是一个推荐的结构，你可以灵活调整章节和篇幅以适应不同的技能类型。核心要求是：**提示词可用的前提下，分析诚实全面。**

通常包含这几个部分：
- **使用说明**（需要准备什么、典型流程、注意事项）
- **完整提示词**（核心产物，代码块包裹）
- **转化分析**（一句话总结 + 实现度 + 保留/丢失的功能 + 手动操作清单 + 体验差异）

分析要做到：
- 实现度有数字，但不过分纠结于精确百分比
- 丢失了什么都说明原因，不是简单列清单
- 需要手动操作的步骤具体可执行，不是"用户自己想办法"
- 体验差异让用户明白使用方式会有多大变化

如果技能**完全无法**迁移（实现度接近 0%），直接诚实告知，不要在提示词部分硬凑。

## 常见陷阱

- **提示词太短**：怕提示词太长用户懒得看，结果把关键规则和示例都砍掉了。用户宁愿复制一个完整的 5000 字提示词，也不想要一个"精简"但不好用的版本
- **分析太模板化**：每个技能的特性不同，分析应该反映出这种不同。不要让每个输出看起来都差不多
- **美化实现度**：觉得说"只实现了 30%"会让用户失望，于是往上报。用户需要的是准确预期，不是安慰
- **忘记用户视角**：始终记住用户要在网页 AI 中使用这个提示词。写完之后自己读一遍提示词，想想如果你是用户，复制到豆包里能不能直接用
